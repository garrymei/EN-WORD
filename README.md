# EN-WORD

EN-WORD 是一个用于背英文单词的网站。核心目标是：用户只需要提供英文单词，系统负责校验、补全释义和学习材料，并通过拼写练习、实时字母反馈和权重调度帮助长期记忆。

## 产品目标

- 支持用户批量导入英文单词。
- 英文单词为必填项，中文释义为可选项。
- 系统自动校验英文单词是否有效。
- 系统自动补全中文释义、音标、发音、短语、例句、翻译和用法讲解。
- 支持配置每日背诵数量。
- 背诵时以“中文释义 + 短语挖空”的方式提示用户拼写英文。
- 拼写过程中实时显示每个字母是否正确。
- 根据拼写过程和最终结果调整单词权重。
- 权重越高，单词出现频率越高。
- 即使单词多次拼写正确，也不会从单词池中移除，只会降低出现频率。
- 长时间没有出现的单词需要逐步提高抽取概率，保证偶尔复现。

## 单词导入需求

用户可以在导入页批量粘贴单词。英文为必填，中文为可选。

支持格式示例：

```text
apple
banana
abandon
```

也支持英文加中文：

```text
apple 苹果
banana 香蕉
abandon 放弃
```

导入时需要执行以下处理：

- 去除每行前后空格。
- 跳过空行。
- 拆分英文单词和可选中文释义。
- 英文单词统一生成 `normalized_word`，用于去重和匹配。
- 校验英文是否为合法单词。
- 校验数据库中是否已经存在该单词。
- 如果用户提供中文释义，需要和系统查询到的中文释义进行比对。
- 保存导入结果，并展示成功、重复、待确认、释义冲突和非法输入数量。

## 英文校验规则

英文单词是必填字段。

校验规则：

- 不能为空。
- 只能包含合理的英文单词字符。
- 需要忽略大小写进行去重。
- 需要查询词典或外部服务确认单词是否真实存在。
- 查询不到的单词不能直接进入正式背诵池，应标记为待确认。

英文校验状态：

```text
valid      已校验，可以进入背诵池
pending    查不到或信息不完整，等待用户确认
invalid    非法输入
duplicate  重复单词，不重复保存
```

## 中文释义规则

中文释义为可选字段。

如果用户提供中文：

- 保存为 `user_meaning`。
- 系统仍然需要查询词典得到 `system_meaning`。
- 如果用户释义和系统释义含义匹配，单词可以进入背诵池。
- 如果用户释义和系统释义明显不匹配，需要标记为释义冲突。

如果用户没有提供中文：

- 使用系统查询到的中文释义。
- 如果系统无法查询到中文释义，则标记为待确认。

释义相关状态：

```text
confirmed  已确认
empty      用户未提供中文，使用系统释义
conflict   用户释义和系统释义冲突
pending    系统释义缺失，等待确认
```

## 自动补全信息

用户只负责提供英文单词，其他信息由系统自动查询、生成并保存。

每个单词需要尽量补全：

- 英文单词
- 标准化英文单词
- 用户提供的中文释义
- 系统查询的中文释义
- 英文释义
- 英式音标
- 美式音标
- 英式发音音频
- 美式发音音频
- 词性
- 常见短语或搭配，优先偏商务和项目管理场景
- 例句
- 例句中文翻译
- 用法讲解
- 数据来源

建议查询策略：

- 先使用词典 API 校验单词，并获取音标、基础释义和发音。
- 再使用 AI 或其他服务生成商务、项目管理相关的短语、例句、翻译和用法讲解。
- 所有查询结果都保存到数据库，背诵时不重复查询。

## 短语与例句需求

背诵时需要优先用短语来承载单词，而不是只展示单词本身。

例如用户只导入：

```text
align
```

系统需要自动生成多个偏商务和项目管理场景的短语：

```text
align with the project goals
align the team on priorities
align expectations with stakeholders
```

背诵时以短语挖空形式出现：

```text
中文：使一致；对齐
短语：_____ with the project goals
```

用户输入完成后，需要展示使用该短语的完整句子：

```text
短语：align with the project goals
例句：We need to align this feature with the project goals before the next milestone.
翻译：在下一个里程碑之前，我们需要让这个功能和项目目标保持一致。
用法讲解：align with 常用于表示计划、功能、团队或预期与目标、战略、优先级保持一致。
```

每个单词应该保存多个短语，短语可以轮换出现。

短语生成要求：

- 优先生成商务、职场、项目管理、会议、需求沟通、交付、风险管理、排期和协作场景中的短语。
- 每个单词至少保存 3 个短语，条件允许时保存 5 到 8 个。
- 每个短语都需要有对应例句、中文翻译和用法说明。
- 短语中的目标单词需要能被挖空，用于拼写练习。
- 短语不应过于生僻，应该贴近日常工作表达。
- 同一个单词的短语要尽量覆盖不同搭配和使用场景。

短语轮换规则：

- 同一个单词每次出现时，优先选择最近较少使用的短语。
- 如果某个短语对应练习中出现过拼写错误，可以适当提高该短语下次出现概率。
- 需要记录每个短语的出现次数和最近出现时间。
- 单词权重决定“哪个单词出现”，短语轮换决定“这个单词用哪个短语出现”。

## 单词池

单词池保存所有已导入单词。单词即使多次拼写正确，也不会被移出池子。

单词池页面需要支持：

- 查看所有单词。
- 搜索单词。
- 按状态筛选。
- 查看用户释义和系统释义。
- 查看音标、短语、例句和用法讲解。
- 编辑单词信息。
- 删除错误单词。
- 手动确认待确认单词。
- 处理释义冲突单词。
- 查看每个单词的权重、正确次数、错误次数和最近出现时间。

## 每日背诵配置

用户可以配置每天背诵数量。

第一版至少支持：

```text
每日背诵数量
```

后续可以扩展：

```text
每日新词数量
每日复习数量
单词抽取模式
是否优先复习高权重单词
```

每天开始背诵时，系统从有效单词池中根据权重抽取指定数量的单词。

## 背诵流程

背诵时，用户不能直接看到完整英文单词。

页面需要显示：

- 中文释义。
- 短语挖空。
- 输入框。
- 字母横杠反馈区域。
- 提交按钮。
- 跳过按钮。
- 发音播放按钮，可选。

示例：

```text
中文：使一致；对齐
短语：_____ with the project goals

请输入单词：
_ _ _ _ _
```

用户提交后展示：

- 正确单词。
- 音标。
- 发音按钮。
- 中文释义。
- 英文释义。
- 本次练习使用的短语。
- 使用该短语的例句。
- 例句翻译。
- 用法讲解。

示例：

```text
align /əˈlaɪn/

释义：使一致；对齐
短语：align with the project goals
例句：We need to align this feature with the project goals before the next milestone.
翻译：在下一个里程碑之前，我们需要让这个功能和项目目标保持一致。
用法讲解：align with 常用于表示计划、功能、团队或预期与目标、战略、优先级保持一致。
```

## 拼写输入反馈

输入过程中，页面下方需要显示和目标单词长度一致的横杠。

例如目标单词是：

```text
apple
```

初始状态显示 5 条灰色横杠：

```text
_ _ _ _ _
```

每条横杠对应一个字母位置。

颜色规则：

```text
未输入：灰色
输入正确：绿色
输入错误：红色
```

判断规则：

```text
用户输入第 n 个字符 == 目标单词第 n 个字符：绿色
用户输入第 n 个字符 != 目标单词第 n 个字符：红色
用户还没有输入第 n 个字符：灰色
```

示例：

目标词：

```text
apple
```

输入：

```text
a
```

反馈：

```text
绿 灰 灰 灰 灰
```

输入：

```text
ap
```

反馈：

```text
绿 绿 灰 灰 灰
```

输入：

```text
aq
```

反馈：

```text
绿 红 灰 灰 灰
```

拼写判断需要忽略大小写。

## 输入过程错误记录

只要用户输入过程中出现过红色横杠，本次练习就需要记录为出现过输入错误。

即使用户后来删除错误字母，并在最终提交时拼写正确，也仍然需要记录：

```text
had_typing_error = true
```

示例：

目标词：

```text
apple
```

用户先输入：

```text
apq
```

第三个位置出现红色横杠。

随后用户删除 `q`，最终输入：

```text
apple
```

虽然最终拼写正确，但本次练习仍然算作“过程中出现过错误”。

## 权重机制

每个单词需要维护基础权重 `base_weight`。

权重目标：

- 拼写越不稳定，权重越高。
- 权重越高，出现频率越高。
- 拼写正确会降低权重。
- 权重不能低于最低值。
- 单词不会因为权重降低而移出池子。
- 很久没出现的单词会获得临时权重加成。

建议默认值：

```text
base_weight 默认值：10
base_weight 最低值：1
base_weight 最高值：100
```

## 权重更新规则

一次练习结束后，根据输入过程和最终结果调整权重。

推荐规则：

```text
全程无红杠 + 最终正确：base_weight - 2
出现过红杠 + 最终正确：base_weight + 1
全程无红杠 + 最终错误：base_weight + 5
出现过红杠 + 最终错误：base_weight + 7
跳过单词：base_weight + 3
```

权重更新后需要限制范围：

```text
base_weight = max(1, min(100, base_weight + delta))
```

## 长时间未出现补偿

为了避免低权重单词长期不出现，抽词时需要根据 `last_seen_at` 计算临时权重加成。

推荐公式：

```text
days_since_seen = 今天 - last_seen_at
overdue_bonus = min(20, floor(days_since_seen / 7))
effective_weight = base_weight + overdue_bonus
```

说明：

- 每 7 天没出现，临时权重加 1。
- 临时加成最高为 20。
- `overdue_bonus` 只用于抽词计算，不一定需要写回数据库。
- 单词一旦被抽中，需要更新 `last_seen_at`。

## 抽词规则

每日背诵时，系统使用 `effective_weight` 进行加权随机抽取。

不是简单选择权重最高的单词，而是按权重计算概率。

示例：

```text
word A effective_weight = 10
word B effective_weight = 20
word C effective_weight = 70
```

总权重：

```text
100
```

出现概率：

```text
word A = 10%
word B = 20%
word C = 70%
```

这样高权重单词更容易出现，低权重单词也仍然有机会出现。

## 学习记录

每次背诵都需要记录学习结果。

需要记录：

- 单词 ID。
- 用户最终输入。
- 是否最终拼写正确。
- 输入过程中是否出现过红杠。
- 权重变化前的值。
- 权重变化后的值。
- 本次权重变化量。
- 是否跳过。
- 背诵时间。

这些记录用于后续统计：

- 哪些词经常拼错。
- 哪些词过程中经常输错但最终能改对。
- 哪些词已经很熟。
- 哪些词很久没有出现。
- 每日正确率。
- 权重变化历史。

## 页面需求

### 单词导入页

- 支持批量粘贴英文单词。
- 支持英文加中文释义。
- 显示导入结果统计。
- 显示重复、非法、待确认和释义冲突的单词。
- 支持对异常单词进行确认、修改或删除。

### 单词池页面

- 查看所有单词。

## 本地运行

安装依赖：

```bash
npm install
```

创建 `.env.local` 并配置数据库连接：

```bash
MYSQL_HOST=
MYSQL_PORT=3306
MYSQL_USER=
MYSQL_PASSWORD=
MYSQL_DATABASE=
```

`.env.local` 已被 `.gitignore` 忽略，不应提交到仓库。

启动开发服务：

```bash
npm run dev -- --hostname 0.0.0.0 --port 3000
```

电脑访问：

```text
http://localhost:3000
```

手机访问时，需要让手机和电脑连接同一个局域网，然后用电脑的局域网 IP 访问：

```text
http://电脑局域网IP:3000
```

如果配置 `.env.local` 中的 `OPENAI_API_KEY`，导入单词时会使用大模型生成更自然的商务和项目管理短语、例句、翻译和用法讲解。未配置时，系统会使用本地兜底内容，网站仍可运行。
